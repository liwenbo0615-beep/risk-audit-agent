## 一、一句话定位

独立设计并实现的多风险域 UGC 内容安全审核 Agent：用 LangGraph 把"识别 → 分析 → 人工复核 → 报告"编排成有状态工作流，采用**离线规则优先、按需升级 LLM** 的两层判定策略，支持 Human-in-the-loop，并对外提供 REST API。

## 二、背景与动机

结合在字节跳动担任大模型应用工程师期间，参与涉政、未成年人等多风险域模型迭代的业务认知，用 Agent 工程化的方式复现内容审核的核心决策链路；目标是做一个结构清晰、可运行、有测试、能体现工程判断力的作品集项目。

## 三、技术栈

- **语言/运行环境**：Python 3.12（conda 环境 `agent_project`）
- **编排框架**：LangGraph 1.x（StateGraph 有状态工作流）、LangChain
- **大模型**：DeepSeek API（`deepseek-chat`，OpenAI 兼容接口）
- **服务化**：FastAPI + Uvicorn（自动 OpenAPI 文档）
- **配置/测试**：python-dotenv、Pydantic、Pytest（44 个单测）

## 四、系统架构

### 4.1 工作流（LangGraph StateGraph，6 节点 + 2 处条件路由）

```
normalize_input          # 文本清洗、截断
   ↓
identify_risk            # 一级风险识别（离线优先 → 按需升级 LLM）
   ↓  route_by_risk      # 路由1：safe 且置信度≥0.75 → 直接出安全报告
   ├─ generate_safe_report → END
   ↓
analyze_risk             # 二级深度分析，定风险等级 + 建议动作
   ↓  route_by_level     # 路由2：按 风险类型/等级/建议动作 决定是否转人工
   ├─ human_review       # Human-in-the-loop（支持自动决策跳过阻塞）
   ↓
generate_report          # 结构化 JSON 审核报告
   ↓
  END
```

- 全局状态用 `TypedDict (RiskState)` 维护，贯穿所有节点，带 `trace_id` 全链路追踪。
- 覆盖 7 类风险（涉政 / 未成年人 / 违法违规 / 暴力 / 垃圾营销 / 安全 / 未知）× 5 个风险等级（high/medium/low/none/unknown）。

### 4.2 核心设计：两层判定 + 按需调用 LLM（项目最大亮点）

`identify_risk` 不是无脑调 LLM，而是：

1. **先跑离线确定性分类器**（关键词规则 + **复合规则**，如"未成年人语境 × 性相关词 → 未成年涉色，命中即高危拒绝"）。
2. **由 `LLMCallJudge` 这个规则驱动的调度器判定该结果是否可信**：
   - 高置信度 / 复合规则命中 → 直接采信离线结果，**不调 LLM**；
   - 涉政等**上下文敏感**类型、"未成年人"等**单关键词易误判**类型、**低置信度**、**unknown** → **升级调用 LLM** 做语义验证。
3. LLM 调用失败/超时 → **自动降级**回退到离线结果，保证无网络/无 Key 也能确定性运行。
4. 每条结果都带 `judge_reason`（如 `offline_sufficient` / `high_confidence_compound_rule` / `escalated:context_dependent:political` / `escalation_failed_offline_fallback`），决策路径完全可追溯、可审计。

> 价值：在保证高危内容召回的前提下，仅对边界/歧义内容才触发 LLM，从而控制调用成本，同时具备离线兜底的鲁棒性。

### 4.3 工程化落地

- **模块化分层**：`classifier`（离线规则）/ `judge`（调度策略）/ `nodes`（图节点）/ `graph`（编排）/ `storage`（日志）/ `config`（环境配置）/ `models`（状态与 Pydantic 模型）职责分离。
- **双运行模式**：`OFFLINE_DEMO_MODE` 一键切换"纯离线（不花钱不联网，结果确定）"与"在线 LLM"。
- **REST API**：`/audit` 单条、`/audit/batch` 批量（单次最多 100 条）、`/logs` 日志查询、`/health` 健康检查；通过线程池避免阻塞事件循环。
- **可观测性**：jsonl 审计日志落盘 + 风险类型/等级分布统计 + 人工复核计数。
- **测试**：44 个 Pytest 单测，覆盖离线分类器、判定调度、图路由、以及 judge 接入主流程后的各条分流路径（含 LLM 失败降级）。

## 七、可能被问到的面试追问与口径

- **"judge 是模型还是规则？"** → 规则驱动的启发式调度器（类型 + 置信度阈值 + 文本长度），不是用小模型路由。讲"规则驱动的升级策略"。
- **"为什么不直接全调 LLM？"** → 成本与鲁棒性：高危内容（违法关键词、未成年涉色复合规则）本地即可高置信判定，没必要付费调 LLM；同时离线兜底保证断网/无 Key 可用。
- **"复合规则解决了什么误判？"** → 例如"未成年人不应该沉迷学习"是安全的，单看"未成年人"关键词会误杀，所以这类交给 LLM 语义判断；而"未成年 × 性相关词"复合命中则是确定高危，本地直接拒。
- **"Human-in-the-loop 怎么落地？"** → 高风险命中人工复核队列，CLI 下交互输入，批处理/API 下用 AUTO_REVIEW_DECISION 自动决策避免阻塞。
